# 产品需求文档：AI 设计诊断工作台

> 文档版本：v1.1
> 创建日期：2026-07-13
> 更新日期：2026-07-13
> 产品类型：Web 应用
> 当前状态：MVP 需求草案
> 核心方向：AI 设计诊断助手

---

## 1. 产品概述

### 1.1 一句话介绍

帮助不会使用专业设计术语的 AI 创作者，把“这张图感觉不对”转化为明确的问题、可执行的修改建议，以及可以直接复制给 AI 的修改 Prompt。

### 1.2 产品背景

AI 已经能够快速生成网页、App、海报和社交媒体图片，但生成结果经常存在排版失衡、视觉层级不清、配色冲突、字体搭配不当或留白不足等问题。

普通创作者通常能感知“差点意思”，却无法准确说明问题所在，只能反复对 AI 说“再高级一点”“配色不好看”“重新生成”，导致迭代效率低、结果不稳定。

本产品的核心不是替用户做设计，而是帮助用户完成：

> 感知问题 → 识别原因 → 准确表达 → 指导 AI 修改 → 验证改进效果

### 1.3 产品愿景

让每个 AI 创作者都能看懂设计问题，并用准确的语言指导 AI 做出更好的设计。

### 1.4 MVP 产品定位

MVP 采用“诊断工作台”形态，优先解决最核心的任务：

> 上传一张设计图，获得具体诊断和修改 Prompt；上传新版后，了解哪些问题改善了、哪些问题仍然存在。

审美训练、提示词训练、知识库和素材库属于产品的后续能力，不作为 MVP 的核心交付范围。

---

## 2. 目标与非目标

### 2.1 MVP 目标

1. 让新用户在 5 分钟内完成第一次设计诊断。
2. 将模糊评价转化为至少 3 条具体、可执行的问题反馈。
3. 为每个重要问题生成可直接复制给 AI 的修改 Prompt。
4. 支持用户上传新版图片，并获得前后版本的改善分析。
5. 验证用户是否愿意持续使用 AI 诊断来辅助设计迭代。

### 2.2 MVP 非目标

- 不提供在线设计编辑器。
- 不在首版直接生成或重绘最终设计图。
- 不把审美训练、系统课程和素材市场作为首版主流程。
- 不承诺 AI 结论绝对正确；诊断结果应作为决策辅助，而不是设计评审的唯一依据。

### 2.3 后续目标

- 通过好坏对比、找茬和维度打分建立审美训练体系。
- 将模糊描述拆解为结构化设计 Prompt。
- 建立设计原则知识库、风格图鉴、配色和字体参考库。

---

## 3. 目标用户

### 3.1 核心用户：AI 设计创作者

- 使用 ChatGPT、Claude、Midjourney、Cursor、Trae 等工具生成设计。
- 能判断“结果不太对”，但不了解对齐、亲密性、层级、对比度等专业概念。
- 需要在短时间内反复迭代网页、App、海报或社交媒体图片。
- 更关心“下一步具体怎么改”，而不是抽象的审美评分。

### 3.2 典型场景

1. AI 生成网页后，用户觉得页面拥挤、重点不突出。
2. AI 生成海报后，用户觉得配色和字体“不高级”。
3. 用户知道想要“左右分割、3:7 比例”，但不知道如何向 AI 表达。
4. 用户根据建议修改后，无法判断新版到底有没有改善。

### 3.3 用户核心任务

> 我不需要成为设计师，但我希望知道问题在哪里，并能准确告诉 AI 怎么改。

---

## 4. 核心用户流程

```text
进入诊断工作台
    ↓
上传图片 / 粘贴图片
    ↓
选择设计类型：界面设计 / 平面设计
    ↓
选择可选诊断重点
    ↓
AI 分析并生成报告
    ↓
查看总体结论、评分和问题清单
    ↓
查看问题原因与修改建议
    ↓
复制单条或完整修改 Prompt
    ↓
在外部 AI 工具中修改
    ↓
上传新版图片
    ↓
查看前后版本改善情况
    ↓
生成下一轮修改 Prompt
```

## 5. 功能需求

### 5.1 诊断工作台（P0）

#### 5.1.1 图片输入

用户可以：

- 上传 JPG、PNG、WebP 图片，单张不超过 10MB。
- 直接粘贴剪贴板图片。
- 拖拽图片到上传区域。
- 预览、替换和删除待诊断图片。
- 为本次诊断填写可选的设计目标，例如“科技感 SaaS 首页”“年轻化活动海报”。

系统需要：

- 校验格式、大小和图片是否可读取。
- 对图片进行压缩或缩放，保证分析效率。
- 明确告知图片的保存期限和删除方式。

#### 5.1.2 设计类型选择

用户必须选择一种诊断模式：

**界面设计模式**

- 页面布局与信息层级
- 网格、对齐与间距
- 组件一致性
- 字体层级与可读性
- 配色与交互元素的视觉权重

**平面设计模式**

- 构图与视觉焦点
- 图文关系与信息层级
- 配色与情绪匹配
- 字体选择与文字可读性
- 留白、密度与画面平衡

#### 5.1.3 诊断重点选择

系统默认分析全部维度；用户可以选择重点分析项：

- 排版与布局
- 配色
- 字体与文字层级
- 留白与视觉平衡

重点分析项用于调整输出优先级，不应导致其他基础问题完全缺失。

### 5.2 AI 诊断报告（P0）

#### 5.2.1 总体结论

报告顶部展示：

- 一句话总结：使用普通用户能理解的语言。
- 整体状态：例如“结构基本清晰，但视觉焦点分散”。
- 4 个维度的 10 分制评分：排版、配色、字体、留白。
- 建议优先处理的前三个问题。
- 诊断置信度和说明：对于图片分辨率不足、文字不可读或信息不完整的情况，必须明确提示。

评分用于帮助用户排序问题，不得单独作为“好看/不好看”的结论。

#### 5.2.2 问题清单

每个问题必须包含：

| 字段 | 要求 |
|---|---|
| 问题标题 | 具体描述，禁止只写“观感不好” |
| 所属维度 | 排版、配色、字体或留白 |
| 问题位置 | 尽可能指出元素、区域或关系 |
| 严重程度 | 高、中、低 |
| 原因解释 | 解释为什么会影响层级、可读性或协调性 |
| 修改建议 | 给出明确动作、方向或比例 |
| 修改 Prompt | 可直接复制给外部 AI |
| 参考原则 | 对比、对齐、重复、亲密性或相关配色原则 |

问题描述示例：

> 页面顶部三个模块的标题字号和字重接近，视觉层级不明显。建议保留一个主标题作为唯一焦点，次级标题缩小至主标题的约 50%–60%，并减少装饰性文字的强调效果。

禁止输出：

- “不够高级”
- “配色不好看”
- “排版有问题”
- 没有说明对象、原因和修改方向的空泛形容词

#### 5.2.3 问题定位

MVP 支持以下两种方式之一，按技术能力择优实现：

1. 在图片上叠加问题编号和大致区域标记。
2. 在问题卡片中使用“页面顶部 / 左侧主视觉 / CTA 区域”等位置描述。

如果 AI 无法可靠定位，不得伪造精确坐标，应降级为文字区域描述。

#### 5.2.4 Prompt 生成

系统支持：

- 复制单个问题的修改 Prompt。
- 复制按优先级合并后的完整 Prompt。
- 在复制前编辑 Prompt。
- Prompt 中包含设计类型、问题位置、修改动作、约束条件和保留项。

Prompt 示例结构：

```text
请保留当前页面的品牌色和主要内容，仅优化布局层级：
1. 将顶部主标题设为页面唯一一级视觉焦点；
2. 将信息模块按 3:7 左右分割布局，不要严格对半；
3. 统一左侧文字与右侧内容的视觉对齐线；
4. 增加模块之间的留白，避免所有元素贴近边界；
5. 不新增装饰元素，不改变核心文案。
```

### 5.3 迭代对比（P0）

用户上传新版图片后，系统需要：

- 关联原始版本和新版版本。
- 对相同维度重新分析。
- 展示改善的问题、未改善的问题和新增问题。
- 对比各维度分数变化，但不将分数变化解释为绝对质量变化。
- 给出下一轮最值得修改的 1–3 个问题。
- 生成下一轮可复制 Prompt。
- 保存版本记录，支持查看历史版本。

对比结果示例：

| 状态 | 示例 |
|---|---|
| 已改善 | 主标题与正文的层级差异更清晰 |
| 部分改善 | 配色更统一，但强调色使用次数过多 |
| 仍存在 | 右侧内容与底部按钮没有形成统一对齐线 |
| 新增问题 | 缩小主视觉后，页面上方出现过多空白 |

### 5.4 迭代修改预判（P1）

用户可以先输入准备交给 AI 的修改意见，系统在实际生成前进行预判：

- 预计改善的问题。
- 可能引入的风险。
- 与当前设计目标是否冲突。
- 更准确的替代表述。
- 一键生成优化后的 Prompt。

### 5.5 诊断历史与反馈（P0）

- 保存最近的诊断任务和版本链路。
- 支持重命名、删除和再次打开任务。
- 用户可对每条建议标记“有帮助 / 没帮助”。
- 用户可说明“不准确的原因”，用于后续质量改进。
- 用户可导出或复制诊断报告。

### 5.6 后续模块路线图

**P1：提示词精准表达**

- 模糊描述拆解为风格、配色、版式、字体、元素、氛围 6 个维度。
- 反向 Prompt 练习。
- 风格 Prompt 模板库。

**P1：审美判断训练**

- 好坏对比题。
- 找茬训练：对比、对齐、重复、亲密性。
- 用户评分与 AI 评分校准。

**P2：设计知识库**

- 四大设计原则。
- 配色安全法则。
- 字体搭配规则。
- 风格图鉴。
- 7 天训练计划。

**P2：资源库**

- 配色推荐。
- 参考图库。
- 字体搭配预览。

## 6. 页面与交互设计

### 6.1 信息架构

```text
首页
├── 立即诊断
├── 诊断历史
├── 提示词训练（P1）
├── 审美训练（P1）
├── 设计知识库（P2）
├── 资源库（P2）
└── 个人中心
```

### 6.2 诊断工作台

- 顶部：任务名称、设计类型、版本切换、重新诊断。
- 左侧：原图 / 当前版本预览及问题标记。
- 中间：总体结论、评分和问题列表。
- 右侧：当前选中问题的原因、建议和 Prompt。
- 底部：上传新版、输入修改方案、进入下一轮对比。

### 6.3 关键状态

必须设计以下状态：

- 未上传图片。
- 上传中。
- 图片格式或大小不符合要求。
- AI 分析中。
- 分析成功。
- AI 分析超时或失败。
- 图片内容无法识别。
- 版本对比成功。
- 诊断次数达到限制。

AI 失败时应保留用户已上传的图片，并提供重试入口；不得丢失任务上下文。

## 7. 非功能需求

### 7.1 性能

- 首屏在常规网络下 3 秒内可交互。
- 图片上传后 2 秒内显示预览和任务状态。
- AI 诊断目标响应时间为 15 秒内；超过 8 秒显示阶段性进度。
- AI 超时后可重试，最多自动重试 1 次，并提供手动重试。
- MVP 支持至少 100 个同时进行的诊断任务，具体容量由压测结果校准。

### 7.2 安全与隐私

- 图片只用于用户发起的诊断，不默认用于训练其他模型。
- 用户可以主动删除图片和诊断任务。
- 服务端保存 API Key，前端不可见。
- 上传文件需校验格式、大小和内容类型。
- 全站使用 HTTPS，用户数据按最小权限原则访问。
- 对第三方 AI 服务的数据传输范围和保存政策进行明确说明。

### 7.3 易用性

- 新用户无需学习设计术语即可完成首次诊断。
- 每个专业词汇都提供通俗解释。
- 诊断结果优先展示“先改什么”，避免一次性堆叠过多信息。
- 桌面端优先，兼容移动端查看和复制 Prompt。
- 所有异步操作有明确的加载、成功和失败反馈。

### 7.4 AI 质量约束

- 输出必须区分事实观察、设计判断和建议，不得把主观判断包装成绝对事实。
- 无法读取文字、颜色或位置时必须明确说明不确定性。
- 不得虚构精确像素、色值或坐标；除非输入信息足以支持该结论。
- 修改建议必须包含对象、动作、方向和限制条件。
- 用户反馈需要进入可追踪的质量评估流程。

## 8. 数据与 AI 评估

### 8.1 核心数据对象

- 用户：账号、订阅状态、使用额度。
- 诊断任务：原图、设计类型、目标描述、创建时间、状态。
- 诊断版本：版本图片、分析结果、评分、生成时间。
- 问题项：维度、位置、严重程度、解释、建议、Prompt、用户反馈。
- Prompt：来源问题、内容、复制次数、用户编辑记录。

### 8.2 AI 质量评估方式

上线前建立一组人工标注样本，覆盖界面设计和平面设计两类输入。至少评估：

- 问题是否具体，而非空泛形容词。
- 问题维度是否归类正确。
- 建议是否可执行。
- Prompt 是否保留用户目标和约束。
- 前后版本是否能识别主要改善项和新增问题。

上线后持续收集：

- 建议有帮助率。
- Prompt 复制率和修改后再次诊断率。
- 用户对“问题准确”的反馈。
- AI 失败率、超时率和单次诊断成本。

## 9. 成功指标

### 9.1 MVP 验证指标

以下指标用于验证产品价值，正式数值应在首轮用户测试后校准：

| 指标 | 目标 |
|---|---:|
| 新用户完成首次诊断比例 | ≥ 60% |
| 首次诊断后复制 Prompt 比例 | ≥ 40% |
| 上传新版进行二次诊断比例 | ≥ 25% |
| 用户认为至少一条建议有帮助 | ≥ 70% |
| 诊断任务成功完成率 | ≥ 95% |
| 用户从上传到看到结果的中位时间 | ≤ 20 秒 |
| 7 日内再次使用诊断比例 | ≥ 25% |

### 9.2 需要谨慎使用的指标

- “AI 诊断准确率”必须先定义人工标注标准后再对外承诺。
- “设计问题描述准确度提升 60%”属于训练模块指标，不应作为 MVP 核心指标。
- 注册量和付费率需要有真实获客渠道与定价实验支持，暂不作为首版验收条件。

## 10. 商业化假设

MVP 阶段只验证使用价值，不急于完整实现支付体系。可采用：

- 免费用户：有限次数的基础诊断。
- 订阅用户：更高额度、版本历史、批量诊断和高级报告。
- 后续再验证团队版、设计评审版和 API 服务。

暂定价格 ¥39/月、¥299/年仅为待验证假设，需通过用户访谈和付费实验确认。

## 11. 风险与应对

| 风险 | 应对方式 |
|---|---|
| AI 输出空泛或不稳定 | 固定诊断结构、建立评估集、收集用户反馈、必要时人工复核样本 |
| 用户把 AI 分数当成绝对标准 | 弱化总分，强调问题优先级和解释依据 |
| 图片无法准确识别文字或布局 | 展示识别限制，允许用户补充设计目标和重点区域 |
| AI 调用成本过高 | 限制图片尺寸、分级模型路由、记录单次成本 |
| 用户上传图片涉及版权或隐私 | 明确用户责任、提供删除能力、限制保存期限 |
| 产品范围不断膨胀 | 以“诊断—修改 Prompt—版本对比”作为 MVP 唯一主闭环 |

## 12. 待确认问题

1. 首版是否允许用户粘贴图片 URL，还是只支持本地上传和剪贴板粘贴？
2. 诊断图片默认保存多久？用户是否可以选择“不保存本次图片”？
3. 首版是否需要登录后才能诊断，还是允许游客体验 1 次？
4. AI 诊断使用哪一家模型，以及单次诊断成本上限是多少？
5. 问题区域标注是 MVP 必须项，还是先使用文字位置描述？
6. 首轮内测计划招募多少名用户，人工评估样本如何建立？

## 附录：诊断输出示例

### 输入

一张 AI 生成的 SaaS 首页截图，用户选择“界面设计”，目标为“专业、清晰、有科技感”。

### 输出

**总体结论**：页面信息完整，但首屏存在多个同等强度的视觉焦点，导致用户不容易判断首先应该看哪里。

**优先问题**：

1. **高优先级｜排版**：主标题、数据卡片和 CTA 按钮的视觉权重接近。建议保留主标题为首要焦点，降低数据卡片的装饰强度。
2. **中优先级｜留白**：模块之间的垂直间距不一致，页面节奏被打断。建议统一主要模块间距，并增加首屏标题与内容区之间的呼吸空间。
3. **中优先级｜配色**：强调色同时用于标题、按钮、图标和边框，视觉注意力被分散。建议将强调色限制在 CTA 和少量关键数据上。

**可复制 Prompt**：

```text
请保留当前 SaaS 首页的内容、品牌色和科技感，仅优化视觉层级与留白：
1. 将主标题设置为首屏唯一的一级视觉焦点；
2. 降低数据卡片和装饰边框的视觉权重；
3. 统一主要模块的垂直间距，增加标题与内容区之间的留白；
4. 将强调色主要保留给 CTA 按钮和关键数据，不要同时用于所有图标和边框；
5. 保持页面整体简洁，不新增装饰元素。
```
